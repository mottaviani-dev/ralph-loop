# Documentation Style Guide

This guide defines tone, structure, and conventions for all project documentation generated by ralph-loop.

---

## Audience & Tone

- **Primary audience**: developers working on or onboarding to this project
- Open with what a feature does for the business (1-2 sentences) before implementation details
- Active voice, present tense: "The auth middleware handles..." not "Handling is performed by..."
- Technical but approachable

---

## Document Structure Template

```markdown
# Feature Name

## Overview
2-4 sentences: what it does, why it matters.

## How It Works
Flow, sequence, key decisions — the core content.
ASCII diagrams encouraged for multi-step processes (keep under 20 lines).

## Key Components

| File | Responsibility |
|------|---------------|
| `path/to/file.ext` | Does X |

## Design Decisions
Why key architectural choices were made. Focus on trade-offs and rationale.

## Related Docs
- [Companion Feature](../category/companion-feature.md)

## Known Gaps
Narrative list of incomplete or unclear areas. Omit if none.
```

Mandatory sections: Overview, How It Works, Key Components, Design Decisions.
Optional sections: Data Flow, Configuration, Known Gaps, Related Docs.

---

## Target Length

- **Module-specific docs**: 150-300 lines
- **Cross-module summaries**: 100-150 lines

If a doc exceeds these limits, split it.

---

## File Path References

- Use **relative paths from the project root**: `src/services/auth.ts`
- **Never** use absolute paths
- Focus on the 3-5 key files — don't catalog every file touched

---

## Code Examples

- Short (5-15 lines), showing only the interesting part
- Omit boilerplate (imports, full component templates)
- Use appropriate syntax highlighting in fenced code blocks

---

## Cross-References & Linking

- Link to related docs using **relative markdown paths**
- Include a **Related Docs** section with 2-5 links when applicable

---

## What NOT to Include

- No YAML frontmatter
- No absolute filesystem paths
- No exhaustive file-to-line-number reference tables
- No speculation about features that "might" exist
- No time estimates or predictions
